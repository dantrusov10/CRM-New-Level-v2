# Отчет о комплексном аудите кода CRM-системы «Решение»

## Summary: Общая оценка здоровья кодовой базы — 4/10

Проект находится в стадии MVP с серьезным техническим долгом. Основные проблемы сосредоточены в области архитектурной целостности, безопасности данных и стабильности процесса сборки. Система сильно завязана на клиентскую логику, что ограничивает её масштабируемость и делает уязвимой для ошибок при параллельной работе пользователей.

---

## Critical Issues: Критические проблемы

1.  **Несобираемый проект (Build Failure):**
    *   В файле `DealDetailPage.tsx` присутствуют ошибки типизации (передача `variant="ghost"` в компонент `Button`, который не поддерживает этот тип). Проект не проходит `npm run build`, что делает невозможным CI/CD.
2.  **Уязвимости в безопасности (PocketBase API Rules):**
    *   Правила доступа (API Rules) в `pb_schema.json` и `collections.json` не согласованы. Многие коллекции имеют слишком широкие права или полагаются исключительно на флаг `role="admin"`, игнорируя контекст владельца данных для менеджеров.
    *   **Риск:** Учетные данные и токены внешних платформ (Тендеры/Парсеры) хранятся в открытом виде в JSON-полях без дополнительного шифрования на уровне приложения.
3.  **Ненадежная инициализация схемы (On-the-fly Schema Init):**
    *   Компонент `DynamicEntityForm.tsx` пытается автоматически создавать разделы и поля при первом рендере.
    *   **Риск:** В условиях конкурентного доступа это приведет к дублированию данных или ошибкам PocketBase "unique constraint". Схема должна управляться миграциями или сидами, а не фронтендом.
4.  **Отсутствие транзакционности:**
    *   Обновление сущности (например, Сделки) и её кастомных полей происходит через несколько независимых HTTP-запросов. Если один из них упадет, данные окажутся в рассогласованном состоянии.

---

## Optimization: Рефакторинг и "мусор"

1.  **Злоупотребление `any` (Type Safety):**
    *   В проекте обнаружено более **140 использований `as any`**. Это полностью нивелирует преимущества TypeScript и скрывает потенциальные ошибки времени выполнения.
2.  **Дублирование логики (DRY):**
    *   В `DealDetailPage.tsx` поля сделки (бюджет, оборот и др.) жестко прописаны в стейте компонента, в то время как `DynamicEntityForm` пытается управлять ими динамически. Происходит "теневое" дублирование данных.
3.  **Мертвый код и зависимости:**
    *   В `package.json` присутствуют уязвимые версии зависимостей (5 уязвимостей, включая Critical).
    *   Отсутствует или не работает конфигурация ESLint (ошибки при запуске `npm run lint`), что приводит к накоплению неиспользуемых импортов и переменных.

## Logic & UX: Архитектурные замечания

1.  **RBAC (Управление доступом):**
    *   Проверка прав реализована преимущественно на уровне UI (скрытие кнопок). В хуках данных (`hooks.ts`) отсутствует централизованная проверка прав, что позволяет получить доступ к данным через консоль браузера, если API PocketBase настроен некорректно.
2.  **Бизнес-логика КП (Калькулятор):**
    *   В `calc.ts` используется округление до 2 знаков на каждой промежуточной строке. При больших объемах позиций это может привести к накопительной ошибке в итоговой сумме. Рекомендуется производить округление только в финале или использовать специализированные библиотеки (decimal.js).
3.  **UX Конструктора полей:**
    *   В админ-панели управления полями (`AdminFieldsPage.tsx`) отсутствует возможность настройки сложных типов (Relations/Options) через UI, хотя движок их поддерживает. Это вынуждает администратора править JSON в базе вручную.

## Предложения по исправлению (Roadmap)

1.  **Phase 1 (Stability):** Исправить ошибки сборки в `DealDetailPage.tsx` и настроить CI с проверкой типов.
2.  **Phase 2 (Security):** Провести аудит всех API Rules в PocketBase. Вынести инициализацию схемы из фронтенда в серверные скрипты миграции.
3.  **Phase 3 (Architecture):** Внедрить Zod для валидации данных с API. Начать планомерную замену `any` на строгие интерфейсы, сгенерированные из схемы PocketBase.
4.  **Phase 4 (DevOps):** Исправить конфигурацию ESLint и обновить уязвимые зависимости.
