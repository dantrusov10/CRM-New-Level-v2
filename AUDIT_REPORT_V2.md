# Отчет о комплексном аудите кода CRM-системы «Решение» (Второй этап)

## Summary: Общая оценка здоровья кодовой базы — 3/10

Несмотря на упоминание проведенных обновлений, текущее состояние кодовой базы остается критическим. Основная причина снижения оценки — **проект всё еще не собирается**, а заявленные ранее проблемы (Build failure, Type Safety, Security) не были устранены. Использование проекта в промышленной эксплуатации (Production) невозможно до исправления ошибок типизации.

---

## Critical Issues: Критические проблемы

1.  **Build Failure (Проект не собирается):**
    *   В файле `DealDetailPage.tsx` сохраняются ошибки: `variant="ghost"` передается в компонент `Button`, который поддерживает только `primary`, `secondary` и `danger`.
    *   **Последствия:** Невозможно запустить `npm run build`, задеплоить приложение или гарантировать работоспособность интерфейса.
2.  **Отсутствие изменений в безопасности:**
    *   API Rules в PocketBase по-прежнему позволяют слишком широкий доступ. В частности, коллекции настроек и парсеров полагаются на проверку `@request.auth.role = "admin"`, но в коде фронтенда (например, `pb.ts`) поле роли считывается как `role_name ?? role`, что указывает на потенциальную путаницу в схемах.
    *   Секреты (токены платформ) в `settings_tender_parser` хранятся в открытом JSON без шифрования.
3.  **Нестабильность DynamicEntityForm:**
    *   Логика `ensureDefaults()` в `DynamicEntityForm.tsx` по-прежнему выполняется на клиенте. При одновременном входе двух пользователей под разными ролями или при медленном соединении возможны конфликты создания полей в PocketBase.

---

## Optimization: Рефакторинг и "мусор"

1.  **Типизация (TS):**
    *   Количество `as any` не уменьшилось (142 вхождения). Это скрывает реальные проблемы с данными, которые приходят из PocketBase.
2.  **Зависимости:**
    *   `npm audit` показывает наличие уязвимостей в зависимостях.
    *   Конфигурация ESLint несовместима с установленной версией (v8 vs v9/v10), из-за чего статический анализ не работает.
3.  **Дублирование данных:**
    *   В `DealDetailPage.tsx` состояние полей сделки (budget, turnover и др.) дублируется между локальным стейтом React и стейтом `DynamicEntityForm`. Это классический источник багов "рассинхрона" данных.

---

## Logic & UX: Замечания по процессам

1.  **Уведомления (NotificationsBell):**
    *   Хук `useMyTasksForBell` запрашивает данные каждые 60 секунд. При росте количества менеджеров это создаст избыточную нагрузку на PocketBase, так как запросы не оптимизированы на уровне индексов (в `pb_schema.json` отсутствуют индексы для полей `due_at` и `is_done`).
2.  **Конструктор полей:**
    *   Админка полей (`AdminFieldsPage.tsx`) все еще не позволяет редактировать `options` для полей типа `select` через интерфейс, что делает "конструктор" бесполезным для бизнес-пользователей без навыков работы с БД.
3.  **Калькулятор КП:**
    *   Логика округления в `calc.ts` не изменилась. Ошибка округления может составлять до нескольких рублей на больших спецификациях.

---

## Резюме:
Обновления, на которые ссылается пользователь, либо не затронули критические зоны, либо не были применены к текущей ветке репозитория. **Первоочередная задача: исправить ошибки в `DealDetailPage.tsx` и восстановить работоспособность `npm run build`.**
