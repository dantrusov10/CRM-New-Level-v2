# AGENTS.md (для AI-агентов / Cursor)

## Кратко о проекте
NewLevel CRM — B2B CRM с воронкой, карточками компаний/сделок, timeline, импортом/экспортом, parser-настройками, модулем КП и заделом под AI decision engine.

## Общие правила работы
1. Сначала изучи текущую архитектуру, потом предлагай изменения.
2. Не ломай бизнес-логику без явного запроса.
3. Не удаляй существующую документацию.
4. Не публикуй секреты и ключи.
5. Любые секреты в текстах помечай как `ENV_SECRET_REQUIRED`.

## Что нельзя ломать
- Авторизация и роли (`users`, `settings_roles`, `rbac.ts`).
- Базовые CRUD потоки по `companies`/`deals`.
- Timeline и связи сущностей.
- Базовую совместимость схемы PocketBase.

## Где искать «правду»
1. `backend/pocketbase/collections.json` — основной bootstrap схемы.
2. `apps/web/src/ui/data/hooks.ts` — фактические вызовы API.
3. `apps/web/src/lib/types.ts` — типы, ожидаемые UI.
4. `README.md` и `/internal_docs/*`.

## Как вносить изменения
- Малые изменения: точечно, без рефакторинга всего проекта.
- Перед изменением схемы: проверить, кто читает коллекцию во фронте.
- После изменений: проверить сборку и ключевые пользовательские сценарии.

## Как работать с PocketBase
- Любая правка схемы должна быть согласована и документирована.
- Не держать несколько конфликтующих «источников истины».
- После schema-правок обновлять документацию в `/internal_docs`.

## Как работать с AI-логикой
- В текущем репо AI runtime ограничен data/UI слоем.
- Если добавляется gateway-интеграция, обязательно:
  - минимизировать данные,
  - валидировать output,
  - логировать безопасно,
  - учитывать лимиты и права.

## Как работать с frontend
- Проверять `router.tsx`, `hooks.ts`, целевые страницы.
- Не нарушать существующий UI-паттерн (`Card`, `Button`, `Input`, `Modal`, `Tabs`).
- Для больших изменений — сначала документировать поведение AS IS.

## Миграции / изменения схемы
- Не смешивать случайно `collections.json` и старые `pb_import_*.json`.
- Любая новая коллекция/поле:
  - описать назначение,
  - описать правила доступа,
  - описать связи,
  - описать план отката.

## Документирование изменений
- Обновлять профильные файлы в `/internal_docs`.
- В changelog/описании фиксировать:
  - что изменено,
  - зачем,
  - риски,
  - что проверить.

## Файлы, которые нельзя трогать без явного разрешения
- `backend/pocketbase/collections.json`
- `apps/web/src/ui/data/hooks.ts`
- `apps/web/src/lib/pb.ts`
- `apps/web/src/router.tsx`
- Любые инфраструктурные конфиги deployment.

## Как проверять результат
- `npm run lint`
- `npm run build`
- Smoke-test:
  - login
  - список сделок
  - карточка сделки
  - timeline
  - admin страницы
  - import/export

## Стиль интерфейса
- Сдержанный B2B UI.
- Высокая читаемость таблиц и форм.
- Предсказуемые действия и явные статусы.

## Стиль кода
- TypeScript first.
- Ясные имена функций/типов.
- Избегать магии и скрытых сайд-эффектов.
- Минимальный, но полезный комментарий.

## Принцип работы
Сначала понять текущую архитектуру и ограничения AS IS, затем менять точечно и безопасно.
