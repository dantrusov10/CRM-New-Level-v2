# CLIENT ONBOARDING AND DB CREATION (AS IS + целевая процедура)

## Важно
В текущем репозитории нет законченного автопровиженинга tenant-инстансов. Ниже — рабочий операционный шаблон «как делать правильно» с учетом текущей структуры проекта.

---

## Базовый поток onboarding
1. Регистрация клиента (коммерческие и юридические данные).
2. Выбор тарифа.
3. Выбор инфраструктурного профиля (shared / dedicated).
4. Создание tenant-среды.
5. Инициализация схемы PocketBase.
6. Создание admin-пользователя клиента.
7. Применение лимитов/feature flags.
8. Выдача URL и доступов.
9. Настройка домена/SSL (если нужно).
10. Настройка AI/парсеров/биллинга.
11. Стартовый импорт данных.
12. Go-live checklist.

---

## A) Минимальный/shared тариф

## Идея
Один инфраструктурный контур обслуживает несколько клиентов, разделение логическое.

## Шаги
1. Создать tenant-запись в control-plane (`tenant_id`).
2. Создать namespace/префиксы доступа и политики rule-level.
3. Применить схему PocketBase (или убедиться, что она уже актуальна).
4. Создать admin пользователя клиента.
5. Назначить тариф и лимиты:
   - users,
   - storage,
   - AI usage.
6. Настроить доступ к frontend (tenant-aware routing).
7. Настроить AI gateway policy для tenant.
8. Выдать инструкции и провести smoke-test.

## Риски shared
- Ошибка в multi-tenant rules может дать утечку данных.
- Сложнее делать per-tenant кастомизации.

---

## B) Максимальный тариф (отдельный VPS)

## Идея
Изолированная инфраструктура: отдельный PocketBase/контур на клиента.

## Шаги
1. Поднять выделенный VPS/instance.
2. Установить PocketBase и базовую ОС hardening-конфигурацию.
3. Применить `collections.json` (или утвержденный export схемы).
4. Создать admin клиента.
5. Настроить домен клиента и SSL.
6. Подключить отдельный AI gateway policy/profile.
7. Настроить backup/restore и мониторинг.
8. Провести acceptance-checklist.

## Плюсы dedicated
- Лучшая изоляция.
- Прозрачная кастомизация и compliance.

## Минусы dedicated
- Выше инфраструктурная стоимость.
- Выше операционная нагрузка.

---

## Копирование схемы PocketBase
- Базовый источник: `backend/pocketbase/collections.json`.
- Команда из `init.sh`:
  - `pb migrate collections import ./collections.json --dir ./pb_data`
- Важно: не смешивать случайно `pb_schema.json` и stage-import JSON без верификации.

## Создание admin-пользователя клиента
- Через PocketBase Admin UI или API.
- Минимум: email + password + role=`admin`.
- Секреты в документацию не писать: `ENV_SECRET_REQUIRED`.

## Применение лимитов
- На текущем этапе это операционная настройка вне этого репо (нет завершенного limit-engine в коде).

## Выдача URL
- Frontend URL + доступы в админку клиента.
- Для dedicated: отдельный поддомен.

## Настройка домена
- DNS + reverse proxy + SSL.

## Настройка AI Gateway
- Проверить разрешенные модели, лимиты, логирование.
- Запретить отправку чувствительных полей во внешние модели.

## Настройка billing
- План, период, usage-метрики, статусы оплаты.
- В текущем репо billing-контур как runtime отсутствует.

## Первоначальные данные
- Импорт компаний/сделок.
- Этапы воронки.
- Пользователи и роли.
- Базовые parser/KP настройки.
