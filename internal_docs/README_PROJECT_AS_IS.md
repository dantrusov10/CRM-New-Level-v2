# NewLevel CRM / CRM «Решение» — AS IS обзор

## Что это за продукт
NewLevel CRM в текущем виде — это CRM для B2B-продаж с упором на управляемый процесс сделки: этапы воронки, ответственность, история изменений, ролевая модель, файловый контур, импорт/экспорт, задел под AI-помощь, парсеры сигналов и генерацию КП.

## Для кого
- Коммерческие команды B2B/enterprise (менеджеры, РОП, админ).
- Компании с длинным циклом сделки и несколькими стейкхолдерами.
- Команды, которым нужен единый контур: сделки + контекст + действия.

## Какую боль решает
- Разрозненные данные по клиенту (таблицы, мессенджеры, заметки).
- Потеря контекста по сделке при смене менеджера.
- Невозможность быстро понять «где риск» и «что делать дальше».
- Слабая управляемость воронки и качества данных.

## Почему это не обычная CRM
- Есть структура под AI-оценки (`ai_insights`) и события AI в `timeline`.
- Есть админ-контур под парсеры (контакты, медиа, тендеры).
- Есть контур КП/спецификации (шаблоны, прайсы, расчеты, PDF).
- Есть RBAC и задел под централизованный control plane.

## Основная ценность
- Единая система фиксации прогресса сделки.
- Прозрачная история действий и решений.
- Подготовка к «decision engine» (score, risks, suggestions, explainability).
- Возможность масштабировать процесс через шаблоны и настройки.

## Основные роли пользователей
- `admin`: полный доступ, настройка системы, пользователи, парсеры, справочники.
- `manager`: операционная работа со сделками и компаниями.
- `viewer`: чтение данных (ограниченный доступ).

## Основные сущности (по фактической схеме)
- База CRM: `users`, `companies`, `deals`, `timeline`, `saved_filters`.
- AI и аудит: `ai_insights`, `audit_log`.
- Парсеры: `parser_runs`, `contacts_found`, `media_items`, `tender_items`, `tender_documents`, `tender_detail_requests`, источники и настройки parser-*.
- Файлы: `files`, `entity_files`, `import_jobs`.
- КП/коммерция: `kp_settings`, `price_lists`, `price_list_items`, `quotes`, `quote_items`, `products`, `product_materials`.
- Роли/карты влияния: `settings_roles`, `role_maps`, `role_map_items`, `semantic_packs`.
- Дополнительно в патче: `tasks`.

## Основные сценарии использования
- Создать компанию и привязать ответственного.
- Создать сделку, вести по этапам, фиксировать комментарии в timeline.
- Смотреть AI-инсайты по сделке (если они созданы внешним контуром).
- Настраивать парсеры и карты ролей (ЛПР/ЛВР/блокеры).
- Собирать КП на основе прайс-позиций, сохранять черновик, выгружать PDF.
- Делать импорт/экспорт данных.

## Текущее состояние проекта
- Frontend: рабочий SPA на React/Vite/TS/Tailwind.
- Backend в этом репозитории: PocketBase schema + скрипты init/run.
- AI Gateway / Control Plane в этом репозитории как исполняемый сервис не присутствует; есть операционный контекст в `AGENT_CONTEXT.md`, что это отдельный контур.

## Что уже реализовано
- Авторизация PocketBase.
- Сделки (таблица/канбан/детальная карточка).
- Компании (список/карточка).
- Воронка и админ-настройки.
- Timeline, файлы, импорт/экспорт.
- Каркас AI-представления в UI через `ai_insights`.
- КП-модуль с расчетом и PDF.

## Что выглядит как задел/каркас
- «AI как decision engine» в полном контуре (gateway, лимиты, биллинг, маршрутизация моделей).
- Multi-tenant SaaS control plane.
- Отдельные dashboard-коллекции (`dashboard_cache`, `dashboard_presets`, `dashboard_views`) — в текущей схеме отсутствуют.
- Некоторые сущности из требований (`deal_field_values`, `company_field_values`, `kp_instances`, `kp_documents`) не совпадают с текущими коллекциями.

## Что требует проверки вручную
- Какая схема сейчас реально «боевая»: `collections.json` vs `pb_schema.json` vs stage-import json.
- Совместимость KPI/КП-модуля с текущими коллекциями (`settings_kp_templates`, `kp_instances` используются во фронте, но отсутствуют в `collections.json`).
- Реальная интеграция AI Gateway и лимитов (в этом репо нет gateway-кода).
- Реальный deployment pipeline backend-контуров (Vercel покрывает только web-часть).
