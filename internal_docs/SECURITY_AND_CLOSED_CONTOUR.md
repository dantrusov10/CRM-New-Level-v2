# SECURITY AND CLOSED CONTOUR (AS IS)

## Какие данные нельзя отправлять во внешние AI
- Персональные данные контактов (телефоны, личные email, мессенджеры).
- Коммерчески чувствительные данные (цены, скидки, условия контрактов).
- Реквизиты и внутренние ID, если это не требуется задаче.
- Любые токены/ключи/пароли.

## Как маскировать данные
- Redaction перед outbound AI-вызовом.
- Псевдонимизация имен и организаций, если возможно.
- Передавать минимально необходимый контекст.

## Как работать с токенами
- Только через env/secret manager.
- Никогда не хранить в коде и публичных JSON.
- В документации указывать `ENV_SECRET_REQUIRED`.

## Где хранить env
- В runtime окружении сервера (CI/CD secrets, server env).
- Локально — в `.env.local`/`.env`, не в Git.

## Что нельзя класть в git
- `.env`, ключи, токены, пароли, приватные сертификаты.
- Экспорт логов с чувствительными данными.
- Реальные production endpoints/credentials.

## Как сделать закрытый контур
- Разместить backend и хранилища в контролируемой инфраструктуре РФ.
- Ограничить outbound трафик и список разрешенных AI endpoints.
- Вести аудит доступа и outbound вызовов.
- Внедрить policy-слой и deny-list полей.

## Как логировать AI-запросы без утечки
- Логировать технические метрики: request_id, tenant/user, модель, latency, usage, статус.
- Не логировать сырой payload без masking.
- Хранить ограниченный retention.

## Как разграничивать доступы
- Ролевой доступ (`admin` / `manager` / `viewer`).
- Owner-based rules для сделок/компаний.
- Отдельные права для админских настроек и аудит-данных.

## Как делать backup
- Регулярный backup PocketBase данных и файлов.
- Тесты восстановления по расписанию.
- Хранение backup в изолированном хранилище.

## Риски AS IS
- В репозитории несколько конкурирующих schema JSON (риск рассинхронизации).
- Часть AI-контуров не реализована как единый защищенный runtime.
- Возможны рассинхроны между фронтом и фактической схемой.
- Нет явного tenant-изоляционного слоя для shared SaaS.

## Что нужно добавить
- Единый security policy документ с обязательным masking pipeline.
- Централизованный secret management.
- Формальная модель данных по классификации чувствительности.
- Автоматический security-check в CI (secret scanning, schema drift checks).
- Регулярный access review по ролям.
